Three Approaches

Pick the architecture that fits — all start with the same BLE reverse engineering phase

A

Web Bluetooth PWA ⭐ Recommended

Stack: TypeScript + React (or plain JS) in the browser

BLE: Chrome's built-in Web Bluetooth API — no WSL/BLE gap, Chrome on Windows talks directly to the adapter

Mobile: Same code runs in Chrome for Android, installable as a PWA

Pros

  • One codebase for desktop + Android
  • No WSL Bluetooth headaches
  • Fast iteration in browser
  • Works today, no build setup needed

Cons

  • Chrome only (no Firefox support for Web BT)
  • Limited background operation on mobile
  • Needs HTTPS for some features (easy to self-sign)
B

Python Desktop → Native Android

Stack: Python + bleak (BLE) + FastAPI backend + simple web UI for desktop, then a separate Android app (Kotlin/Java) later

BLE: Python bleak running on Windows (not WSL), serves data via local HTTP to the browser in WSL

Pros

  • Python is great for rapid BLE prototyping
  • bleak is a mature BLE library
  • Full background operation on desktop

Cons

  • WSL can't access BLE — Python must run on Windows side
  • Two separate codebases for desktop + mobile
  • More moving parts
C

Flutter Cross-Platform

Stack: Dart + Flutter + flutter_blue_plus for BLE

Builds to: Windows desktop app + Android APK from one codebase

Pros

  • True native UI on both platforms
  • Excellent BLE support via flutter_blue_plus
  • Single codebase, real app experience

Cons

  • Need to learn Flutter/Dart if unfamiliar
  • Heavier build toolchain to set up
  • Slower initial iteration

Phase 0 (same regardless of choice)

BLE reverse engineering: nRF Connect scan → Android HCI snoop log → protocol documentation
